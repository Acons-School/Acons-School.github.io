ACONS Information System Case Study

1. Data Handling & Registration

The system begins by collecting student data. A person registers by providing their Full Name, Age, Address, and License Type. Our system performs a front-end validation check: if the user is between 16 and 18, the system flags a mandatory Parental Permission requirement.

2. Examination Phases

Once registered, the data follows the student through two distinct phases:

  • Theoretical Phase: Tracking the "Code" learning progress and exam results.
  • Practical Phase: Managing driving hours, "Circuit" training, and final road exams.

3. Conceptual Data Model (CDM)

This model illustrates the high-level relationships between Students, Instructors, and Exam Appointments.

4. Logical Data Model (LDM)

Our LDM defines the specific relations and attributes that structure the ACONS database. We have identified the following relational schema:

  • CATEGORY (category_id, category_code, minimum_age)
  • TRAINEE (trainee_id, first_name, last_name, dob, address, phone, status, #category_id)
  • INSTRUCTOR (instructor_id, first_name, last_name, phone)
  • LESSON (lesson_id, type, date, time, status, duration, #instructor_id)
  • EXAM (exam_id, type, date, status)
  • ATTENDS (#trainee_id, #lesson_id)
  • TAKES (#lesson_id, #exam_id)

Note: Underlined attributes are Primary Keys; attributes preceded by # are Foreign Keys.

5. SQL Implementation

The final stage is implementing the schema into an SQL database to ensure persistent storage and easy retrieval.

CREATE TABLE CATEGORY ( category_id INT PRIMARY KEY, category_code VARCHAR(2), -- A, B, C, D minimum_age INT ); CREATE TABLE TRAINEE ( trainee_id INT PRIMARY KEY AUTO_INCREMENT, first_name VARCHAR(50), last_name VARCHAR(50), dob DATE, phone_number VARCHAR(15), category_id INT, FOREIGN KEY (category_id) REFERENCES CATEGORY(category_id) );
